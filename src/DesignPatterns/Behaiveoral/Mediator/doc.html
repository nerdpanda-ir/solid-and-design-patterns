• Mediator:

    هدف این دیزاین پترن ایجاد یک راه برای interact و communication آبجکت ها با یکدیگر ، بدون داشتن رابطه با هم میباشد !!!

    این design pattern در جایی استفاده میشود که چندین object باید همدیگر را بشناسند و بهم اشراف داشته باشند و با هم رابطه داشته باشند و یکدیگر را در خود compose کنند تا بتوانند با هم interact و communication کنند !!!

    مثلا برنامه نویس client ما در کد نهایی ، object هایی دارد که انها برای interact و Communication کردن با هم ، باید یکدیگر را بشناسند و به هم اشراف داشته باشند !!!

    اما شناساندن همه ی object ها به یکدیگر در کدنهایی ، ممکن است کاری پیچیده ، سخت ، هزینه بر و زمانبر باشد و برنامه نویس client ما را درگیر پیچیدگی ایجاد روابط بین object ها بکند و همچنین قطعا انجام این کار ، کدنهایی ما را شلوغ و پیچیده و کثیف میکند !!!

    این مشکلات را میتوانیم به واسطه دیزاین پترن mediator حل کنیم به این صورت که این design pattern مارا مجبور میکند تا یک unit بسازیم و سپس آن object هایی را که میخواهند با یکدیگر interact و communicate داشته باشند را به این unit بشناسانیم و معرفی کنیم

    اینطوری دیگر نیاز نیست ان object ها برای communicate و interact باهم ، یکدیگر را بشناسند و با هم اراتباط داشته باشند و برنامه نویس ...

    | اینطوری دیگر نیاز نیست ....| سپس هر object ای که میخواهد با object های دیگر communicate و interact کند ، میتواند درخواست خود را به یونیت mediator بدهد و چون یونیت mediator به همه ی object ها اشراف دارد و همه ی object ها را میشناسد و با همه ی object ها ارتباط دارد آن درخواست را تحویل object مورد نظر میدهد !!!! | اینطوری دیگر نیاز نیست object هایی که میخواهند با هم communicate و interact کنند همدیگر را بشناسند و به هم اشراف داشته باشند و با هم ارتباط داشته باشند و استفاده از این design pattern باعث کاهش روابط بین object هایی که میخواهند با هم communicate و interact کنند میشود !!! | --- بواسطه دیزاین پترن مسعولیت communicate و interact دیگه نیاز نیست به یکدیگر بشناسانیم وفقط به مدیاتور میشناسانیم

    برای حل این مشکل میتوانیم از دیزاین پترن Mediator استفاده کنیم و communicate و interact کردن آن object ها با یکدیگر را بر عهده این design pattern بسپاریم !!! | به این صورت که این design pattern ما را مجبور میکند تا object هایی را که میخواهند با یکدیگر communicate و interact کنند را به mediator معرفی کنیم و بشناسانیم حالا هر object ای که میخواهد با object های دیگر communicate و interact کند ، میتواند درخواست خود را به mediator بدهد و به دلیل اینکه mediator به همه ی object ها اشراف دارد و همه ی object ها را میشناسد و با همه ی object ها ارتباط دارد آن درخواست را تحویل object مورد نظر میدهد !!!! | | |

    اینطوری به جای اینکه برنامه نویس Client ما در کدنهایی object ها را به یکدیگر بشناساند و درگیر پیچیدگی ایجاد روابط بین ان object ها شود میتواند هر object را فقط به یونیت Mediator بشناساند و معرفی کند ، حالا هر object ای که بخواهد با یک object دیگر communication و interact کند ، بدون شناخت همدیگر و داشتن رابطه متقابل با هم ، میتواند درخواست خود را به یونیت Mediator بدهد و چون یونیت Mediator همه ی object ها را میشناسد و به همه object ها اشراف دارد آن درخواست را تحویل object مورد نظر میدهد !!!

    استفاده از دیزاین پترن Mediator باعث شده برنامه نویس client ما در کد نهایی دیگر درگیر شناساندن object ها به یکدیگر نشود
