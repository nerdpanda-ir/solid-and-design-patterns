هدف این دیزاین پترن Encapsulate متد هایی که بر روی یک object پردازش و یا عملیات انجام میدهند و سپس ان object را تغییر میدهند و یا manipulate میکنند در class های مجزا میباشد !!!

این Design Pattern در جایی استفاده میشود که یک class دارای method هایی باشد که آن method ها بر روی object هایش یک پردازش و یا عملیات انجام میدهند و سپس ان object را تغییر میدهند و یا آن را manipulate میکنند !!!

مثلا در سیستم یک object داریم که برنامه نویس client ما بواسطه method هایی که درون آن object وجود دارد بر روی ان object یک پردازش و یا عملیات انجام میدهد و سپس آن object دستکاری میشود و یا تغییر میکند اما وجود چنین method هایی در class آن object که بر روی آن object یک پردازش و یا عملیات انجام میدهند و سپس انرا دستکاری میکنند و یا آنرا تغییر میدهند باعث شده تا تعداد method های آن class بیش از حد زیاد شود و ساختار و code آن class نیز پیچیده ، شلوغ و کثیف شود و همچنین ممکن است آن method ها خارج از وظیفه ی آن class باشند و وجود آن method ها باعث شوند که دیگر آن کلاس single responsible نباشد و به علاوه برنامه نویس client ما نمیتواند به راحتی و به آسانی و به صورت Dynamic چنین method هایی را به ان class اضافه کند زیرا open / close principle در آن class نقض شده در نتیجه تمامی اینها باعث شده تا کار با ان class برای برنامه نویس client ما پیچیده ، ‌سخت و دشوار شود
